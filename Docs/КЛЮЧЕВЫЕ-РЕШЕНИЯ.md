# Ключевые и самые сложные решения

Документ фиксирует основные архитектурные решения и наиболее тонкие места реализации
(потоки, уплотнение окон, сидеры ограничений). Полезно для защиты и для будущих правок.

---

## Самые основные решения

### 1. Солвер работает с потоками, а не с группами
Оси модели и все ограничения (пересечения, окна, дневные лимиты, переходы между корпусами)
строятся вокруг `AcademicStream → StreamGroups → Group → Course → Degree → Institute`. Группа
— это участник потока, а единица планирования — поток. Это позволяет ставить общую лекцию один
раз на несколько групп.

### 2. Потоки выводятся из по-групповых данных ММИС при загрузке
ММИС (как и реальный «Деканат») хранит учебный план **по группам**: лекция каждой группы —
отдельная строка. Потоки не приходят из источника, а **вычисляются в коде** при синхронизации
по ключу **(курс, дисциплина, преподаватель)** при `LessonType = Лекция`. Семинары и
лабораторные остаются по группам. См. `StreamComposition`.

### 3. Декомпозиция генерации + жёсткие блокировки занятых ресурсов
Монолитная модель института не помещалась в память (OOM/502), поэтому институт разбивается на
компоненты связности по общим группам, каждый решается отдельной небольшой моделью, а уже
занятые `(аудитория, слот)`/`(преподаватель, слот)` передаются следующим как жёсткие блокировки.

### 4. Пост-уплотнение убирает «окна» после генерации
Окна возникают из-за последовательной декомпозиции (слот «застолблён» ранее обработанной
подзадачей), а штраф за окно мягкий и не пересиливает жёсткую блокировку. Поэтому окна убираются
отдельным детерминированным проходом, переставляющим уже размещённые пары. См. `ScheduleCompactor`.

### 5. Сидеры идемпотентны и детерминированы
Всё, чего нет в ММИС (аудиторный фонд, календарь, оборудование, веса, ограничения), наполняется
сидерами с детерминированными Guid и гейтингом по наличию — повторный прогон не плодит дублей.
Ограничения, зависящие от данных ММИС (оборудование на планах, доступность, корпус), сидятся
**после** синхронизации.

## Самые сложные решения

### A. Слияние лекций в потоки внутри журналируемого идемпотентного диффа
Файлы: `Infrastructure/Mmis/StreamComposition.cs`, `MmisSyncService.cs`.

Сложность в том, что синхронизация ММИС — идемпотентный upsert с журналированием нагрузки
(add/update/delete по детерминированным Guid). Слияние меняет кардинальность: N по-групповых
лекций должны стать одним планом и одной нагрузкой, иначе солвер поставит лекцию N раз.

Принятая схема:
- Лекции группируются по `(курс, дисциплина, преподаватель)`; объединяются только наборы из
  ≥ 2 групп.
- Выбирается «первичный» участник (минимальный MMIS-id) — стабильный детерминированный выбор.
- Создаётся **план потока** с Guid из ключа слияния; его поток связывается со всеми группами,
  численность = сумма групп.
- Нагрузка **первичного** участника перенаправляется на план потока (репойнт `CurriculumId`),
  нагрузки остальных участников **поглощаются** (не создаются).
- Поглощённые по-групповые планы не попадают в `seen` → штатная логика удаляет их наследие
  из БД с журналированием. План потока — в `seen`, поэтому стабилен между синками.

Тонкость: на первом синке поверх старой (1:1) БД нужно одновременно создать план потока,
перенаправить нагрузку первичного участника и удалить старые по-групповые планы/нагрузки — без
нарушения внешних ключей (репойнт в фазе 1, удаление в фазе 2). Покрыто `StreamCompositionTests`
и проверено вживую: 504 строки ММИС → 372 плана (66 потоков по 3 группы).

### B. Пост-уплотнение с сохранением всех жёстких ограничений
Файл: `Application/Solver/Compaction/ScheduleCompactor.cs`.

Перенос пары в более ранний слот не должен нарушить ни одно жёсткое ограничение:
- пересечения ресурсов (аудитория/преподаватель/группа), включая блокировки **чужих институтов**;
- смена группы (первая смена — пары 1–4);
- **переходы между корпусами** — проверяются по соседним *существующим* слотам дня (как в солвере),
  а не по номеру пары;
- двойные пары вообще не двигаются, чтобы не разорвать спаренность.

Перенос принимается, только если он **строго уменьшает** суммарное число окон у затронутых
субъектов (преподаватель + группы). Это одновременно решает две задачи: гарантирует прогресс к
цели (меньше окон) и **завершимость** алгоритма (монотонно убывающий потенциал). Реализован
консервативный вариант: пара остаётся в своей аудитории (сохранены корпус, вместимость,
оборудование). Покрыто `ScheduleCompactorTests`.

### C. «Вечерние» преподаватели — мягкая модель из-за жёсткой смены
Файл: `Infrastructure/Seed/InfrastructureSeeder.cs` (`SeedTeacherAvailabilityAsync`).

Группы в dev-данных — первой смены (жёстко пары 1–4). Жёсткий «только вечер» сделал бы расписание
неосуществимым (мягкое не пересиливает жёсткую смену, лекции остались бы в 1–4 с огромным штрафом).
Поэтому «вечер» моделируется **мягко** — штрафом `Discouraged` за ранние пары (1–2): солвер
сдвигает занятия к поздним из доступных пар, осуществимость сохраняется. Жёсткий вариант
возможен при появлении вечерних групп — отмечено в коде.

### D. Сид потоков на уровне курса при сохранении осуществимости
Файл: `mmis-mssql/sql/02_seed.sql`.

Чтобы лекции **совпадали** у групп одного курса (и сливались в поток в коде), дисциплина и лектор
задаются на **уровне курса**, а не группы (раньше выбирались по индексу группы — лекции не
совпадали, потоки не формировались). Семинары и лабы — по группам, нередко с другими
преподавателями. Объёмы подобраны так, чтобы поток (≤ ~90 студентов) помещался в поточный зал
(120) и расписание оставалось осуществимым; точные часовые «цели» преподавателей сознательно
ослаблены в пользу читаемости и появления потоков. Проверено прогоном в одноразовой БД:
все 66 лекционных ключей охватывают ровно 3 группы.

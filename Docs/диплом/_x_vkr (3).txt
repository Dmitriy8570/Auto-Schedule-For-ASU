Существующие решения для создания форм, такие как Google Forms, Microsoft Forms или Typeform, представляют собой преимущественно инструментальные платформы, ориентированные на создание и распространение форм в рамках закрытых экосистем. Кроме того, такие решения характеризуются недостаточной гибкостью в настройке интерфейса, ограниченными аналитическими возможностями и зависимостью от внешних сервисов. После выбора технологий необходимо приступить к проектированию структуры базы данных, что является одним из ключевых этапов разработки любого информационного приложения. Данный процесс включает в себя детальный анализ предметной области, выявление основных бизнессущностей, определение их атрибутов и установление логических связей между ними.
В контексте объектно-реляционного моделирования сущность представляет собой абстрактный объект реального мира, обладающий определенными характеристиками и свойствами, которые имеют значение
28

для разрабатываемой системы. Каждая сущность характеризуется набором атрибутов (полей), которые описывают ее свойства и хранят конкретные значения данных. Атрибуты могут быть простыми (неделимыми) или составными (состоящими из нескольких компонентов), обязательными или необязательными, а также могут служить в качестве первичных или внешних ключей для обеспечения целостности данных.
В рамках разрабатываемого приложения каждая выявленная сущность будет реализована в виде C# класса, где атрибуты сущности представлены как свойства (properties) класса с соответствующими типами данных. Entity Framework Core использует подход "Code First", при котором структура базы данных определяется на основе классов-сущностей, что обеспечивает тесную интеграцию между объектной моделью приложения и реляционной структурой базы данных.
Центральным компонентом архитектуры данных выступает класс контекста базы данных WoasFormsDbContext, который наследуется от IdentityDbContext<WoasFormsAppUser>. Данный класс служит точкой входа для всех операций с базой данных и содержит DbSet<T> свойства для каждой сущности, обеспечивая объектно-реляционное отображение (ORM).
Особое значение в Entity Framework Core имеют навигационные свойства (navigation properties) - специальные свойства сущностей, которые представляют связи с другими сущностями. Навигационные свойства позволяют осуществлять навигацию по объектному графу, обеспечивая прямой доступ к связанным данным без необходимости выполнения дополнительных запросов к базе данных. EF Core поддерживает различные типы связей: один-к-одному (one-to-one), один-ко-многим (one-to-many) и многие-ко-многим (many-to-many).
29

Навигационные свойства могут быть определены как у одной из связанных сущностей (односторонняя навигация), так и у обеих сущностей (двусторонняя навигация), что обеспечивает гибкость в проектировании модели данных и упрощает реализацию бизнес-логики приложения. EF Core способен автоматически обнаруживать и конфигурировать связи на основе соглашений, анализируя навигационные свойства и их типы.
Дополнительная конфигурация связей и ограничений может быть выполнена в методе OnModelCreating класса WoasFormsDbContext с использованием Fluent API, что позволяет точно настроить поведение Entity Framework Core в соответствии с требованиями предметной области. Данный подход обеспечивает полный контроль над генерируемой схемой базы данных.
Важным преимуществом Entity Framework Core является его способность к автоматическому созданию и обновлению структуры базы данных посредством механизма миграций (migrations). Система миграций отслеживает изменения в модели данных и генерирует соответствующие SQL-скрипты для применения этих изменений к физической базе данных, обеспечивая версионирование схемы данных и упрощая процесс развертывания приложения в различных средах.

2.4 Проектировка необходимых страниц В контексте разработки современных веб-приложений с использованием фреймворка ASP.NET Core Blazor, фундаментальным принципом архитектурного проектирования является создание четко структурированной иерархии пользовательского интерфейса, основанной на логическом разделении функциональных областей приложения.
33

Согласно парадигме компонентно-ориентированной разработки, каждому логически обособленному разделу приложения должна соответствовать отдельная страница, выполняющая специфические функции отображения релевантных данных и, при необходимости, обеспечивающая механизмы приема и обработки пользовательского ввода. Данный подход способствует поддержанию принципов единственной ответственности (Single Responsibility Principle) и разделения интересов (Separation of Concerns), что является краеугольным камнем качественной архитектуры программного обеспечения.
Платформа ASP.NET Core Blazor предоставляет разработчикам расширенный инструментарий для создания пользовательских интерфейсов, включающий в себя два основных архитектурных элемента: razor-страницы и изолированные компоненты.
Razor-страницы представляют собой полнофункциональные вебстраницы, которые служат основными точками входа для различных разделов приложения. Каждая страница инкапсулирует в себе как логику представления данных, так и механизмы взаимодействия с пользователем, обеспечивая целостный пользовательский опыт в рамках конкретной функциональной области. Особенностью архитектуры ASP.NET Core Blazor является наличие специализированной директивы @page, которая существенно упрощает и автоматизирует процесс настройки системы маршрутизации. Данная директива позволяет декларативно определить URLадреса, по которым будет доступна конкретная страница, устраняя необходимость в ручной конфигурации маршрутов в отдельных файлах настроек. Применение директивы @page обеспечивает тесную интеграцию между логикой представления и правилами навигации, что способствует
34

созданию более согласованной и легко сопровождаемой архитектуры приложения.
Параллельно с созданием полноценных страниц, фреймворк предоставляет возможность разработки изолированных компонентов — модульных элементов пользовательского интерфейса, которые можно многократно использовать в различных контекстах приложения. Эти компоненты функционируют как переиспользуемые строительные блоки, позволяющие разработчикам создавать сложные пользовательские интерфейсы путем композиции более простых, специализированных элементов.
Применение данной архитектурной стратегии обеспечивает ряд существенных преимуществ:
• Масштабируемость: Модульная структура облегчает расширение функциональности приложения без нарушения существующей архитектуры.
• Сопровождаемость: Четкое разделение ответственности между компонентами упрощает процессы отладки, тестирования и модификации кода.
• Переиспользование: Компонентный подход способствует созданию библиотек переиспользуемых элементов интерфейса, что повышает эффективность разработки.
• Тестируемость: Изолированные компоненты легче поддаются модульному тестированию, что способствует повышению качества программного продукта.
Таким образом, архитектурная модель ASP.NET Core Blazor, основанная на сочетании страничной структуры и компонентного подхода,

Глава 3. Реализация приложения 3.1 Реализация сущностей приложения. Миграция базы данных.
В рамках реализации слоя доступа к данным был создан класс WoasFormsDbContext, который наследуется от встроенного IdentityDbContext<WoasFormsAppUser>. Данный подход позволяет интегрировать систему управления пользователями ASP.NET Core Identity с пользовательской моделью данных приложения. Внутри контекста в методе OnModelCreating() выполняется явная настройка отношений между сущностями с использованием Fluent API, что обеспечивает точное определение типов связей (один-к-одному, один-ко-многим, многие-комногим) и дополнительных ограничений на уровне базы данных.
Классы сущностей представляют собой гибридные модели, содержащие как примитивные свойства (строки, числа, даты), так и навигационные свойства для связанных объектов. Навигационные свойства играют ключевую роль в процессе создания миграций Entity Framework Core, автоматически определяя структуру внешних ключей и индексов. В дальнейшем эти свойства значительно упрощают разработку сервисов приложения, предоставляя fluent-интерфейс для работы с данными через LINQ-запросы, что делает работу с базой данных намного удобнее по сравнению с прямым обращением к БД через SQL-запросы.
3.2 Реализация сервисов приложения
В архитектуре приложения реализованы три основных сервиса, которые выполняют роль слоев абстракции между пользовательским интерфейсом и внешними системами. DatabaseAccessorService служит промежуточным слоем между Razor-страницами и Entity Framework Core, инкапсулируя логику работы с базой данных и предоставляя упрощенный интерфейс для выполнения CRUD-операций. JiraTicketsService обеспечивает
40

взаимодействие с REST API Jira для управления тикетами техподдержки, скрывая сложность HTTP-запросов и сериализации данных за простыми методами. SFContactManager аналогично реализует слой абстракции для работы с REST API Salesforce CRM, предоставляя удобные методы для управления контактами.
Все три сервиса регистрируются в контейнере внедрения зависимостей и внедряются только в те Razor-страницы, где их функциональность действительно необходима. Такой подход соответствует принципу разделения ответственности и позволяет избежать избыточных зависимостей, делая код более поддерживаемым и тестируемым. Каждая страница получает доступ только к тем сервисам, которые требуются для реализации её конкретной логики.
3.3 Реализация razor страниц приложения Пользовательский интерфейс приложения реализован с использованием Razor-страниц, исходный код которых представляет собой смесь HTML-разметки и минимального количества C#-кода. C#-код выполняет роль связующего звена, вызывая соответствующие функции внедренных сервисов для получения и обработки данных.
Такая архитектура обеспечивает четкое разделение презентационной логики и бизнес-логики, при этом сама страница остается максимально простой и фокусируется на отображении данных и обработке пользовательского ввода. Сервисы для интеграции с внешними платформами реализованы с использованием различных методов аутентификации в зависимости от требований каждой системы. JiraTicketsService использует базовую аутентификацию с логином и паролем для доступа к REST API Jira, что позволяет выполнять операции создания, чтения и обновления тикетов техподдержки. SFContactManager реализует более сложную схему аутентификации через OAuth2 токены для работы с Salesforce CRM, включая сервис управления токенами для автоматического обновления истекших токенов доступа. Все API-ключи и учетные данные для доступа к внешним сервисам внедряются из системных файлов конфигурации и никогда не сохраняются в репозитории исходного кода. Конфиденциальная информация хранится исключительно на машине-сервере в защищенных конфигурационных файлах, что обеспечивает безопасность доступа к внешним системам и соответствует лучшим практикам разработки.

3.5 Доставка приложения на облачный сервис, конфигурация облачной машины. Настройка доменного имени
Для обеспечения публичного доступа к разработанному вебприложению было выполнено его развертывание в облачной инфраструктуре Oracle Cloud Infrastructure (OCI). В рамках данного этапа была создана виртуальная машина на базе операционной системы Linux Ubuntu, которая предоставила необходимые вычислительные ресурсы для функционирования приложения в продакшн-среде.
Процесс развертывания включал несколько ключевых этапов. Первоначально приложение было скомпилировано в исполняемый формат для Linux-платформы с использованием команды dotnet publish с соответствующими параметрами кроссплатформенной компиляции. Полученные исполняемые файлы, а также копия базы данных SQLite были переданы на удаленный сервер посредством SSH-протокола, обеспечивающего безопасную передачу данных.
На виртуальной машине был установлен и настроен веб-сервер Nginx, выполняющий функции обратного прокси-сервера для перенаправления HTTP-запросов к ASP.NET Core приложению. Конфигурация Nginx включала настройку проксирования запросов на локальный порт, на котором функционирует Blazor-приложение. Для обеспечения автоматического запуска и управления жизненным циклом приложения был создан системный сервис Linux (systemd service), который автоматически запускает приложение при старте системы и перезапускает его в случае сбоя, передавая необходимые параметры конфигурации и переменные окружения.
Все операции по конфигурированию сервера, установке зависимостей, настройке сервисов и управлению файлами выполнялись через SSH-консоль,

что обеспечило безопасный удаленный доступ к серверу и возможность администрирования без графического интерфейса.
Завершающим этапом развертывания стала регистрация доменного имени через сервис Namecheap по специальной акционной цене $1 на годовой период. Интеграция доменного имени с Nginx потребовала соответствующих изменений в конфигурационных файлах веб-сервера для корректной обработки запросов к указанному домену.

## 1. Окружение и как технически работать с файлами

- **Рабочая папка** на этом ПК: `C:\Users\barda\OneDrive\Desktop\диплом` (на другом устройстве
  путь может отличаться, напр. `C:\Users\bardakov\Desktop\Диплом` — это нормально, файлы те же).
- **Правка `Диплом.docx`** — через **python-docx** (нужен Python 3 и `pip install python-docx`).
  **pandoc отсутствует.** Для больших русских текстов писать отдельный `.py`-скрипт и запускать
  его (надёжнее, чем `python -c` с кириллицей в командной строке). Скрипты называть с префиксом
  `_` и удалять после использования.
- **Чтение PDF** (образцы ВКР, требования) — `pdftotext -enc UTF-8 in.pdf out.txt` (есть в
  Git Bash / `/mingw64/bin`), затем читать txt. Прямое чтение PDF как изображения не используем.
- **Рисунки/схемы** — рисую через **PIL (Pillow)**, шрифт `C:/Windows/Fonts/arial.ttf` (кириллица
  поддерживается). `matplotlib` и `graphviz` НЕ установлены (при желании можно поставить — дадут
  лучшее качество 3D и графов).
- **Кодировка консоли Windows = cp1251.** Прямой вывод кириллицы и спецсимволов (`≈ ∈ Σ →`) в
  stdout падает с `UnicodeEncodeError` или выдаёт кракозябры. Решение: писать результат в
  UTF-8-файл и открывать его, либо печатать только ASCII.
- **Word блокирует файл.** Если `Диплом.docx` открыт в Word, рядом появляется `~$Диплом.docx`, и
  запись через python-docx даёт `PermissionError`. Перед сохранением — закрыть файл в Word.

### Параметры оформления (нормоконтроль АлтГУ + единообразие с введением)
- Шрифт Times New Roman 14 пт, межстрочный интервал 1.5, абзацный отступ 1.25 см
  (`Emu(450215)` в python-docx), основной текст по ширине.
- **Заголовок главы:** по центру, полужирный, ПРОПИСНЫЕ, с новой страницы
  (`page_break_before`). Формат «`1 НАЗВАНИЕ`» — БЕЗ слова «ГЛАВА» и без точки после номера
  (так в образцах ВКР кафедры; методичка допускает и «ГЛАВА 1» — если нормоконтроль потребует,
  поменять одну строку).
- **Подразделы:** «`1.1 Название`» — полужирный, первая буква прописная, без точки в конце.
- **Подпись таблицы:** «Таблица 1.1 — …» 12 пт, по правому краю, НАД таблицей; в тексте
  обязательна ссылка; глава/раздел не должны заканчиваться таблицей (после неё — текст).
- **Подпись рисунка:** «Рисунок 1.1 — …» 12 пт, по центру, ПОД рисунком; ссылка в тексте до
  появления рисунка.
- **Ссылки на литературу:** числовые в квадратных скобках `[N]`, сквозные по порядку первого
  упоминания (ГОСТ Р 7.0.5-2008). Список литературы — после заключения, по ГОСТ 7.1-2003,
  30–50 источников, русские источники раньше иностранных.

## 4. Классификация ограничений солвера (ФИНАЛ, подтверждена автором 2026-06-08/09)

**Жёсткие (4)** — обязательны, в модели как ограничения:
1. запрет пересечений по ресурсам (аудитория / преподаватель / поток);
2. точное соответствие объёму нагрузки (Σ назначенных пар = Hours/2);
3. соответствие оборудования (аудитория обязана иметь требуемое дисциплиной оборудование);
4. двойные занятия (лабораторные и т. п. — два смежных слота, `Curriculum.Double`).

**Мягкие (7)** — штрафы в целевой функции `F(x)=Σ p_k·n_k(x)→min`:
1. «окна» у студенческих групп и преподавателей (`WindowSectionBuilder`, StudentGap/TeacherGap);
2. **доступность преподавателей и аудиторий — 3 градации времени** «запрещённое / нежелательное /
   предпочтительное» с убывающими весами штрафа (НЕ бинарно!);
3. **смена обучения группы** — штраф растёт по мере отклонения слота от предпочтительного времени
   начала смены (напр., у 2-й смены занятия тяготеют к 5-й паре);
4. **перемещение между корпусами** — большой штраф за смежные занятия одного потока/преподавателя
   в разных корпусах при нехватке времени на переход;
5. предпочтительный корпус (`Curriculum.FavoriteBuilding`);
6. **параллельность подгрупп** — поощрение параллельной постановки, когда разрешена планом
   (`Curriculum.Parallelism` — булев флаг «можно параллельно»; необязательно);
7. **лимит пар в день** — мягкий с большим штрафом (преп. 7, группа 4; в коде пока жёсткий, помечен
   комментарием «НЕОБХОДИМО ИЗМЕНИТЬ НА СИСТЕМУ ШТРАФОВ»).

Доп. слагаемых целевой функции автор добавлять НЕ просил — только эти. Веса штрафов сейчас
зашиты в коде, в перспективе — настройка администратором.

### Что реализовано в коде солвера (на момент изучения, ветка dev-v1)
`Application/solver/`: `ScheduleModel` (`Lessons[w,r,t]` BoolVar, CP-SAT), `ScheduleData`,
`ScheduleModelDirector` + `IModelSectionBuilder` (builder-паттерн, расширяемость). Билдеры:
`VariablesSectionBuilder` (переменные), `IntersectionSectionBuilder` (пересечения),
`TotalHoursConstraintSectionBuilder` (часы), `DailyLessonsLimitSectionBuilder` (лимит, пока
жёсткий), `WindowSectionBuilder` (окна). Остальные ограничения спроектированы в домене (поля
есть), билдеры автор допишет. Домен: `Curriculum` (Parallelism, Double, FavoriteBuilding,
LessonType, NeededEquipments), `SemesterWorkload/WeekWorkload`, `Lesson` (ScheduleVersion
Current/Draft), `Availability` (Teacher/Classroom, поле Penalty), `Constraints` (enum
ConstraintType: TeacherGap, StudentGap, ClassroomAvailability, TeacherAvailability), логи
(`SemesterLog/WeekLog`).

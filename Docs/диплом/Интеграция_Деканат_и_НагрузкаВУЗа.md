# Интеграция с ММИС: «Деканат» + «Нагрузка ВУЗа»

> Рабочая заметка по проектированию выгрузки справочников и нагрузки из систем ММИС.
> Цель: получить группы/преподавателей из «Деканата» и часы из «Нагрузки ВУЗа»,
> корректно сопоставив их между собой. Решения по реализации — на потом, здесь зафиксирована
> аналитика и конкретный план.

---

## 1. Краткий вывод

- Твоя интуиция верна: **«Деканат» — справочник (мастер-данные) групп, преподавателей,
  кафедр, учебных планов; «Нагрузка ВУЗа» — источник часов**. Выгружать списки групп и
  преподавателей именно из «Деканата» правильнее.
- Оба продукта — **из одной интегрированной линейки ММИС Лаборатории**, спроектированы
  работать вместе. Справочники в «Нагрузке» по большей части **приходят из «Деканата»**.
- **Главное правило сопоставления: соединять по внутренним кодам/ID ММИС, а не по ФИО и
  названиям групп.** Записи нагрузки почти наверняка ссылаются на код преподавателя и код
  группы/плана, и эти коды обычно те же, что в «Деканате».
- Одна ли это физическая БД — **не гарантировано**, нужно уточнить у администратора (бывает и
  раздельно с обменом, и общий MS SQL-инстанс).

---

## 2. Экосистема ММИС и кто чей источник правды

ММИС Лаборатория (г. Шахты) — это набор интегрированных модулей с общими справочниками:

| Модуль | За что отвечает | Что отдаёт нам |
|---|---|---|
| **Планы (ГОСИНСП / Планы ВПО)** | учебные планы, дисциплины, виды занятий | дисциплины, виды нагрузки |
| **Деканат** | контингент студентов, движение, группы, иногда штат преподавателей | **группы, курсы, кафедры, преподаватели** |
| **Нагрузка ВУЗа** | расчёт и распределение нагрузки кафедр/преподавателей | **часы (семестровые/недельные)**, привязка преподаватель↔дисциплина↔группа |
| Расписание / Эл. ведомости / Личный кабинет | потребители результата | — |

Поток данных внутри ММИС, как правило:
**Планы → Деканат → Нагрузка ВУЗа**. То есть «Нагрузка» сама импортирует контингент и планы
из «Деканата», поэтому коды сущностей в ней — это коды «Деканата».

**Назначение источников для нашей системы:**
- Группы / курсы / контингент → **Деканат** (первично).
- Кафедры / преподаватели → **Деканат** (часто; в некоторых внедрениях штат правится в
  «Нагрузке», это надо проверить — см. чек-лист).
- Дисциплины / учебные планы → **Планы/Деканат**.
- Часы нагрузки → **Нагрузка ВУЗа** (первично).

## 3. Хранение и развёртывание: одна БД или несколько

Нельзя считать, что это одна база. Реально встречается:

1. **Раздельные БД** (исторически локальные DBF/Paradox/FoxPro, в современных версиях —
   MS SQL). Обмен между «Деканатом» и «Нагрузкой» — через **встроенный механизм обмена ММИС**
   (выгрузка/загрузка файлов). Тогда у нас два источника, и сводить их надо у себя.
2. **Общий MS SQL-сервер**, несколько баз (а иногда модули делят одну базу). Тогда возможен
   прямой `JOIN` между таблицами «Деканата» и «Нагрузки» на стороне SQL.

Практическое следствие: архитектуру интеграции надо строить так, чтобы **работала в обоих
случаях** — то есть через явный слой сопоставления по внешним ключам, а не в расчёте на
один `JOIN`. Если окажется, что база одна и коды общие — слой просто выродится в тривиальный
lookup.

## 4. Сопоставление сущностей — конкретно

### 4.1. Основной путь: по кодам ММИС

ММИС внутри использует числовые/строковые коды для кафедры, преподавателя, группы,
дисциплины. Поскольку «Нагрузка» наполняется из «Деканата», в записях нагрузки преподаватель
и группа — это **коды, совпадающие с «Деканатом»**.

Алгоритм:
1. Выгрузить из «Деканата» преподавателей и группы вместе с их **кодами ММИС**.
2. Выгрузить из «Нагрузки» строки нагрузки — там преподаватель/группа тоже представлены
   **кодами**.
3. Соединить по коду. Никакой эвристики, сопоставление детерминированное.

Что проверить заранее (см. чек-лист §8): действительно ли в таблицах нагрузки лежат коды
(а не текст), и равны ли они кодам в «Деканате».

### 4.2. Запасной путь: по натуральным ключам (хрупкий)

Если общих кодов нет (раздельные внедрения, ручной ввод) — матчинг по атрибутам. Он
ненадёжен, применять только вынужденно и с ручным разрешением конфликтов.

**Преподаватели** — составной ключ:
- Нормализованное ФИО: `UPPER`, `TRIM`, схлопнуть двойные пробелы, `ё → е`, привести
  инициалы к единому виду («Иванов И. И.» / «Иванов Иван Иванович»).
- + кафедра (код/название) — снимает однофамильцев.
- + по возможности табельный номер.
- Проблемы: однофамильцы («Иванов И.И.» ×3), смена фамилии, опечатки, совместители на
  нескольких кафедрах.

**Группы** — составной ключ из компонентов кода группы:
- код направления/специальности + год набора + номер группы + (форма обучения).
- Проблемы: переименование/переформирование групп между семестрами, слияния/деления,
  подгруппы.

Вывод: натуральный матчинг ведём через **таблицу соответствий** с возможностью ручной
правки, а не «на лету».

## 5. Как это ложится в нашу архитектуру

Clean Architecture, адаптер интеграции живёт в **Infrastructure**, контракты — в **Application**.

### 5.1. Изменения в доменной модели

Добавить на импортируемые сущности внешний ключ источника, чтобы повторные выгрузки
**обновляли**, а не плодили дубли, и чтобы привязывать нагрузку по стабильному ключу:

```
// концептуально, на Teacher / Group / Department / Course / Subject / Curriculum
public string? ExternalId { get; private set; }     // код в ММИС
public ExternalSource Source { get; private set; }  // Dekanat | NagruzkaVuza | Manual
```

- Уникальный индекс `(Source, ExternalId)`.
- При натуральном матчинге — отдельная сущность связки:
  `ExternalEntityLink { EntityType, LocalId, Source, ExternalId, ConfidenceLevel }`.

### 5.2. Контракты в Application

Разделить **источник справочников** и **источник нагрузки** — они физически разные системы:

```
public interface IReferenceDataSource          // Деканат
{
    Task<IReadOnlyList<ExternalTeacher>>    GetTeachersAsync(CancellationToken ct);
    Task<IReadOnlyList<ExternalGroup>>      GetGroupsAsync(CancellationToken ct);
    Task<IReadOnlyList<ExternalDepartment>> GetDepartmentsAsync(CancellationToken ct);
    Task<IReadOnlyList<ExternalCurriculum>> GetCurriculaAsync(CancellationToken ct);
}

public interface IWorkloadSource               // Нагрузка ВУЗа
{
    Task<IReadOnlyList<ExternalWorkload>> GetWorkloadsAsync(Guid semesterId, CancellationToken ct);
}
```

`External*` — это «сырые» DTO источника, обязательно с полем `ExternalId`/кодами связей
(`TeacherExternalId`, `GroupExternalId`, `SubjectExternalId`).

### 5.3. Реализация адаптеров в Infrastructure

- `DekanatReferenceDataSource` и `NagruzkaWorkloadSource` — отдельные read-only подключения
  (`Microsoft.Data.SqlClient` или отдельные `DbContext` к MS SQL).
- Только `SELECT`, отдельная учётка с минимальными правами, чтение со
  `READ COMMITTED SNAPSHOT` / снапшот-изоляцией (или с реплики), чтобы не лочить прод ММИС.

### 5.4. Сервис синхронизации (ETL)

Фоновая задача (`IHostedService` / Hangfire / Quartz), период из проекта — **сутки или 3
часа**:

1. **Импорт справочников из «Деканата»** → upsert по `(Source=Dekanat, ExternalId)`:
   кафедры → преподаватели → курсы → группы → дисциплины/планы.
2. **Импорт нагрузки из «Нагрузки ВУЗа»** → для каждой строки нагрузки найти локальные
   `Teacher`/`Group`/`Subject` **по ExternalId**; если не найдено — в карантин/лог рассинхрона,
   не падать.
3. **Детекция изменений** → при изменении часов писать `SemesterLog`/`WeekLog`
   (это и есть наша фича «история изменений нагрузки»).
4. Запись в нашу PostgreSQL в транзакции **SERIALIZABLE** (защита от коллизий — как заявлено).

Идемпотентность: всё через upsert по внешнему ключу; повторный прогон не создаёт дублей.

### 5.5. Конвейер (схема)

```
Деканат (MS SQL) ──IReferenceDataSource──┐
                                         ├─► SyncService ─► upsert(ExternalId) ─► PostgreSQL
Нагрузка ВУЗа (MS SQL) ─IWorkloadSource──┘        │
                                                  └─► diff часов ─► SemesterLog/WeekLog
```

## 6. Детекция изменений для истории нагрузки

Чтобы наполнять `WeekLog`/`SemesterLog` (Add/Update/Delete), на каждой синхронизации:
- сравнивать новое значение часов с текущим в нашей БД по ключу
  `(CurriculumExternalId, период)`;
- при расхождении — писать запись лога с `OldValue`/`NewValue`/`TimeStamp`;
- удаление строки нагрузки в источнике → `LogAction.Delete`.

Если в «Нагрузке» есть собственное поле версии/даты изменения (`timestamp`/`дата правки`) —
использовать его для **инкрементальной** выгрузки (брать только изменённое), иначе сравнивать
снапшоты.

## 7. Риски и митигации

| Риск | Митигация |
|---|---|
| Нет общих кодов между системами | Слой `ExternalEntityLink` + матчинг по натуральным ключам с ручным разрешением |
| Однофамильцы среди преподавателей | Ключ ФИО + кафедра (+ табельный номер) |
| Переименование/переформирование групп | Хранить историю кодов групп, не терять связь по `ExternalId` |
| Схема ММИС недокументирована и меняется на обновлениях | Читать через согласованную **VIEW/контракт**, а не сырые таблицы; адаптер изолирует изменения |
| Нагрузка на прод ММИС | Снапшот-изоляция / чтение с реплики; периодичность 3 ч–сутки |
| Частичная рассинхронизация (нагрузка ссылается на отсутствующий справочник) | Карантин строк + лог рассинхрона, не падать целиком |
| Доступ к чужой прод-БД | Только read-only учётка, согласование с ИТ-управлением |

## 8. Чек-лист вопросов администратору ММИС

1. «Деканат» и «Нагрузка ВУЗа» — на одном MS SQL-инстансе или раздельно? Версии СУБД/продукта?
2. В таблицах нагрузки преподаватель/группа/дисциплина — это **коды (ID)** или текст?
3. Совпадают ли эти коды с кодами в «Деканате» (общие справочники)?
4. Где первичен список **преподавателей** — «Деканат» или «Нагрузка»? А кафедры?
5. Есть ли штатный механизм **выгрузки/обмена** ММИС (XML/файлы/веб-сервис), которым можно
   пользоваться вместо прямого чтения таблиц?
6. Есть ли в таблицах поля **даты/версии изменения** (для инкрементальной выгрузки)?
7. Можно ли получить **read-only доступ** (учётку) или **реплику/выгрузку**, а не доступ к
   боевой базе напрямую?
8. Документирована ли схема (ER, словарь таблиц), или нужен реверс?

---

## 9. Для разработки без доступа к боевым системам

- Поднять **MS SQL в Docker** (`mcr.microsoft.com/dotnet`/`mcr.microsoft.com/mssql/server`),
  создать схемы-заглушки «как Деканат» и «как Нагрузка», залить представительные тестовые
  данные с общими кодами — и отладить весь конвейер импорта и сопоставления.
- В проде заменить только строки подключения адаптеров.

## 10. Ближайшие конкретные шаги (когда вернёшься к теме)

1. Получить ответы по чек-листу §8 (особенно п.2–3 — есть ли общие коды).
2. Добавить `ExternalId` + `Source` в доменную модель и миграцию.
3. Описать контракты `IReferenceDataSource` / `IWorkloadSource` в Application.
4. Сделать заглушки MS SQL в Docker и тестовый набор данных.
5. Реализовать `SyncService` (upsert по ExternalId + детекция изменений → логи).
6. Реализовать адаптеры чтения из реальных БД, подменив заглушки.

## 1. «Информация о проекте» (со слов автора, дословно)

Ссылка на проект на github: https://github.com/Dmitriy8570/Auto-Schedule-For-ASU/tree/dev-v1.
Проект ещё находится на начальной стадии разработки. Моя часть проекта на ветке `dev-v1`.

Я Бардаков Дмитрий Александрович, группа 4.205-2. Научный руководитель Михеева Т.В., к.т.н.,
доцент кафедры информатики. Название диплома: «Разработка веб-приложения для автоматизированного
формирования учебного расписания АлтГУ».

Проект командный, команда из двух человек. Я разрабатывал бэкенд-часть приложения. Товарищ
занимался frontend-ом и docker.

**Причина создания приложения.** Составлением расписания в АлтГУ занимается бюро расписаний —
они составляют всё расписание для всего университета вручную и вводят все данные в Excel-таблицу
с макросами. Для этого мы ходили к ним узнать, как происходит составление расписания. Изначально
они пишут всё на бумаге, а потом заполняют таблицу в Excel и выгружают это всё. Они работают с
приложением **НагрузкаВУЗа**, где есть вся информация о нагрузке каждого преподавателя в неделю.
Также у них есть список кабинетов. Была попытка перехода на «1С:Университет», но это приложение
не соответствовало их требованиям, поэтому от него отказались.

**Выбор технологий.** Так как автоматическое составление расписания — NP-трудная задача, для
модуля автоматического составления была выбрана библиотека **Google OR-Tools**, способная решать
задачи программирования в ограничениях с помощью решателя **CP-SAT**. Библиотека есть на Java,
C++, Python и C#. Выбран **C#**. Для БД выбран **PostgreSQL**.

## 2. Текст к защите (формулировки, годятся для текста диплома)

> Файл-источник: `repo/Docs/Текст к защите_Бардаков.docx`. Использовать как готовые
> формулировки. Заявленное здесь описывается в дипломе как реализованное (см. правило в CLAUDE.md).

**Актуальность.** Составление учебного расписания — одна из ключевых административных задач
любого университета. С точки зрения теории это NP-трудная комбинаторная задача, в которой нужно
одновременно учесть сотни ограничений: доступность преподавателей, аудиторный фонд, смены
обучения, требования к оборудованию. В АлтГУ процесс ведётся преимущественно вручную с
использованием **макросов VBA, разработанных в начале 2000-х**. Составление расписания занимает
несколько недель, любые изменения публикуются с задержкой в несколько дней, а при параллельном
редактировании файлов возникают коллизии, которые крайне сложно отследить.

**Цель.** Спроектировать серверную часть системы — разработать структуры данных и алгоритмическое
обеспечение для автоматической генерации учебного расписания.

**Проблемы текущего процесса:** отсутствие нормализованной реляционной модели данных (→
пересечения занятий при параллельном редактировании); устаревший инструментарий (локальные
Excel-таблицы и VBA-макросы — риск потери данных, нет поддержки); Excel не позволяет применять
современные методы комбинаторной оптимизации; низкое качество результата — «окна» у студентов и
преподавателей, нерациональное использование аудиторного фонда.

**Обзор аналогов:** отечественные «1С:Расписание» и «Апекс-ВУЗ», зарубежные **UniTime** и **FET**.
Ни одно решение не интегрируется с ММИС и не адаптировано под регламенты университета — это
обосновывает разработку собственной специализированной системы.

**Формализация ограничений (как заявлено к защите):**
- **Жёстких — 7:** запрет пересечений ресурсов, доступность преподавателей и аудиторий,
  соответствие оборудования, смена обучения группы, лимит пар в день, время перемещения между
  корпусами.
- **Мягких — 2:** параллельность подгрупп, постановка лабораторных работ двойными занятиями.
- Архитектура модуля расширяема: добавление нового правила не требует изменения существующего кода.

> Примечание: формулировка «7 жёстких / 2 мягких» — из речи к защите. В рабочих заметках
> (CLAUDE.md) разбиение чуть иное (лимит пар в день → мягкое с большим штрафом; «окна» и
> предпочитаемый корпус — мягкие). При написании опираться на CLAUDE.md, эту версию держать как
> вариант формулировки.

**Стек:** C# + ASP.NET Core (сервер), PostgreSQL + EF Core (хранение), Google OR-Tools (алгоритм),
Чистая архитектура (расширяемость и тестируемость).

**База данных:** три предметные группы — аудиторный фонд, учебная нагрузка, расписание. Ключевые
решения: транзакции **SERIALIZABLE** (исключают коллизии при параллельной записи), миграции схемы
(версионируемость структуры БД).

**Алгоритм:** Google OR-Tools, решатель CP-SAT. Для каждого сочетания «занятие — временной слот —
аудитория» создаётся булева переменная. Жёсткие ограничения — запреты, мягкие — штрафы в целевой
функции. Решатель применяет распространение ограничений + поиск с возвратом + эвристики. Результат —
расписание с минимальным числом нарушений мягких ограничений при 100% соблюдении жёстких.

**Тестирование (как заявлено):** модульные тесты xUnit покрывают каждое ограничение; интеграционные
тесты проверяют взаимодействие БД, REST API и модуля OR-Tools. На тестовом наборе алгоритм
сформировал расписание с нулём нарушений жёстких ограничений. Транзакционная модель PostgreSQL
подтвердила корректность при параллельных запросах.

**Заключение (результаты):** проведён анализ предметной области и формализована система ограничений;
спроектирована нормализованная БД PostgreSQL; разработан доменный слой на C# с Чистой архитектурой;
реализован модуль автоматической генерации расписания на Google OR-Tools.

**Зачем.** Солвер уже потребляет `TeacherAvailability`/`ClassroomAvailability` (поле `Penalty`),
`Curriculum.{Parallelism,Double,FavoriteBuildingId,NeededEquipments}`, `Group.Shift`. Но задать
их через API/UI нельзя. Это ядро ценности продукта (правила вуза, которых не было в «1С») и
напрямую экран «Ограничения»/`PropertiesManager` из макета.

### A0. Доменные заметки и развилки (решить до реализации)
- **Availability моделирует только «штраф за нежелательность»** (`Penalty`), без положительного
  «предпочтения». UI макета даёт 5 градаций (обязат./предпочт./нейтр./нежел./запрещено).
  Развилка:
  - (рекомендую) Маппинг градаций в один знаковый вес: запрещено → большой штраф, нежел. → малый,
    нейтр. → запись отсутствует, предпочт. → отрицательный штраф (бонус), обязат. → очень большой
    бонус. Тогда `Penalty` может быть знаковым — проверить, что builder'ы это допускают
    (`AvailabilitySectionBuilder`).
  - Либо хранить только 3 «отрицательных» градации (запрещено/нежел./нейтр.), а «предпочт./обязат.»
    не поддерживать в первой версии (упростить UI). Решить с автором.
- **`Group.Shift` не nullable** (First/Second/Evening) — в UI есть «не указана». Либо добавить
  значение `Unspecified`, либо трактовать «не указана» как «без штрафа смены».
- Все доменные поля — `private set`. Нужны **доменные методы-мутаторы** (напр.
  `Curriculum.SetParallelism/SetDouble/SetFavoriteBuilding/SetEquipment`,
  `Teacher.SetAvailability(...)`), а не правка через рефлексию.
- Availability привязан к (день, пара) **без недели/семестра** — глобальная доступность. Уточнить,
  достаточно ли (вероятно да: доступность преподавателя не меняется по неделям).

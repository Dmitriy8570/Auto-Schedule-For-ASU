# Что доделать до готового продукта — полный разбор

Составлено после подробного изучения ВКР-документов (`Docs/диплом/`: ИСХОДНИКИ-проекта,
Вопросы для дипломной работы, Интеграция_Деканат_и_НагрузкаВУЗа) и всего кода
(`backend-auto-schedule/src`, `Frontend/src`). Это не только список багов — это карта того, чего
**не хватает как продукту**, исходя из задумки ВКР и реальных болей бюро расписаний.

## Боли бюро (из ВКР) → закрыто ли

| Боль бюро расписаний | Статус в коде |
|---|---|
| Одновременная работа без защиты от коллизий | ❌ **не закрыто** (нет блокировок, владельца черновика, real-time) |
| Пересечения отслеживаются вручную | ⚠️ только при автогенерации; ручные правки коллизии не проверяют |
| Изменённое расписание публикуется через 3 дня | ⚠️ публикация есть, но без real-time и без изоляции пользователей |
| Нет сервиса уведомлений об изменении нагрузки | ❌ **не закрыто** (есть только лог-история, без уведомлений) |
| Автосоставление с правилами вуза | ✅ солвер CP-SAT + ограничения реализованы |
| Интеграция с ММИС («Нагрузка ВУЗа») | ⚠️ каркас синхронизации есть, но без `ExternalId`/сопоставления и реальной БД |

Легенда приоритетов: 🔴 ядро/критично · 🟡 важно для полноты · 🟢 качество/эксплуатация.

---

## 1. 🔴 Совместная работа без коллизий — ЯДРО ЦЕННОСТИ (сейчас не реализовано)

Это главная причина существования продукта, и именно её в коде нет.

- **У занятия нет владельца.** `Lesson` не хранит, кто его создал (нет `CreatedByUserId`).
  Из-за этого невозможен заявленный в ВКР сценарий: «занятия пользователя помечаются Draft →
  кнопка *Сохранить* публикует их → *Отмена* удаляет **все Draft этого пользователя**».
  Сейчас Draft/Current — глобально по институту, черновики разных людей не изолированы.
  → Добавить владельца занятия + команды «опубликовать мои черновики» / «отменить мои черновики».
  _Файлы: `Domain/schedule/Lesson.cs`, новые команды, `LessonRepository`._

- **Публикация и генерация — глобальные, перетирают друг друга.** `PublishInstituteScheduleCommand`
  переводит в Current **все** Draft института (без семестра и без пользователя) и удаляет прежний
  Current; `GenerateInstituteScheduleCommand` удаляет всё расписание института. Двое сотрудников,
  работающих по одному институту/в один семестр, затрут результаты друг друга.
  → Скоупить по семестру и пользователю; запретить публикацию чужих черновиков.
  _Файлы: `PublishInstituteScheduleCommand.cs`, `GenerateInstituteScheduleCommand.cs`,
  `LessonRepository.GetByInstituteAsync` (без фильтра семестра)._

- **Нет транзакций SERIALIZABLE / оптимистичной блокировки.** В ВКР заявлено как реализованное
  решение коллизий, но в коде изоляция нигде не задаётся (только MMIS-sync открывает обычную
  транзакцию). Конкурентные генерация/публикация/правки — гонки без защиты.
  → Обернуть мутации расписания в `SERIALIZABLE` (с ретраем при сериализационном сбое) и/или
  добавить токен конкуренции (`xmin`/`RowVersion`) на `Lesson`.
  _Файлы: команды мутаций, `ApplicationDbContext`/конфигурации._

- **Нет real-time синхронизации.** Бюро — 7 человек; они должны видеть изменения друг друга сразу
  (а не «через 3 дня»). Нет SignalR/WebSocket.
  → Добавить SignalR-хаб: рассылка обновлений расписания + «мягкие блокировки» редактируемого
  объекта (кто-то редактирует группу/преподавателя — других предупреждаем).

## 2. 🔴 Уведомления об изменении нагрузки (заявленная главная боль — нет сервиса)

Бюро вручную сверяет нагрузку, потому что «нет сервиса, который уведомляет об изменении».
В коде есть **лог** изменений (`SemesterLog`/`WeekLog`, вкладка «История»), но нет **уведомлений**.

- → При синхронизации с ММИС, если изменились часы плана, по которому **уже стоит занятие в
  расписании** (связь `Lesson.CurriculumId` теперь есть), уведомлять ответственного: лента/
  «колокольчик» в UI (+ опционально e-mail). Показывать, какие занятия затронуты.
  _Файлы: `Infrastructure/Mmis/MmisSyncService.cs`, новая сущность/таблица уведомлений, SignalR/UI._

---

## 3. 🔴 Роли и зона ответственности

- **Ролей нет** — любой аутентифицированный (член группы `ScheduleBureau`) может всё: генерация,
  публикация, удаление занятий, CRUD аудиторий/оборудования, правка весов. По ВКР роли две:
  администратор и сотрудник бюро.
  → Ввести роли (claim в JWT) + `[Authorize(Roles=...)]` на админских/мутирующих эндпоинтах.
  _Файлы: `Infrastructure/Auth/JwtTokenService.cs`, `LdapAuthenticationService.cs`, контроллеры._

- **«За каждым закреплены свои институты»** — этой привязки нет в модели. Сотрудник должен править
  расписание только своих институтов.
  → Смоделировать связь пользователь→институты и ограничивать доступ.

## 4. 🟡 Ограничения, которые нельзя задать из UI (солвер их учитывает, но данных нет)

- **Доступность преподавателей (`TeacherAvailability`) и аудиторий (`ClassroomAvailability`).**
  Домен и builder в солвере есть (`AvailabilitySectionBuilder`), но **нет ни API, ни UI** — данные
  ввести нельзя, ограничение фактически не работает. Это и есть назначение вкладки «Ограничения»
  из макета (сетка доступности по дням/парам).
  → CRUD доступности + редактор-сетка во фронте.

- **Оснащение аудиторий оборудованием (`EquipmentRoom`) и требования занятия (`NeededEquipment`).**
  Оборудование теперь создаётся/удаляется, но **не связывается** с аудиториями и планами →
  ограничение оборудования в солвере не наполняется.
  → Связывание «аудитория ↔ оборудование» и «план ↔ требуемое оборудование» + UI.

- **Параметры плана** (предпочитаемый корпус, двойная пара `Double`, параллельность `Parallelism`,
  смена) задаются только из ММИС/сидов — нет ручной правки из UI (на усмотрение, но для полноты).

- Веса мягких ограничений (`ConstraintConfig`) — ✅ уже сделано (CRUD + вкладка «Веса»).

## 5. 🟡 Полнота работы с расписанием

- **Проверка коллизий при ручном добавлении/перемещении пары** — нет (можно поставить две пары в
  одну аудиторию/преподавателю/группе в один слот). Добавить проверку + уникальные индексы БД.
  _Файлы: `CreateLessonCommand.cs`, `LessonConfiguration.cs`._
- **Редактирование существующей пары** (`PUT /lessons/{id}`) — нет (только add/delete/move).
- **Генерация «по вузу» и «по преподавателю»** — нет, только по институту (в UI заглушки).
- **Публикация/откат** — подключить к кнопкам «Выгрузить»/«Сбросить до выгруженного», скоупить по
  семестру и пользователю (см. п.1).
- **Экспорт расписания (Excel/PDF)** — бюро сейчас выгружает результат в Excel; в продукте нужен
  экспорт/печать по институту/группе/преподавателю (замена их Excel-выгрузки).
- **Просмотр всего расписания** (не только по одному объекту) для печати/проверки.

## 6. 🟡 Справочники, аудиторный фонд, календарь

- **CRUD аудиторий и оборудования** — ✅ сделано.
- **Семестры и недели** — есть чтение (`/semesters`, `/weeks`), но **нет создания** семестра и
  генерации его недель (Red/Blue, даты). Без этого не с чем работать. Добавить управление.
- **Корпуса/институты/кафедры/группы/преподаватели** — только чтение (ожидается из ММИС). Пока
  ММИС не подключён к реальной БД, нужен либо **ручной CRUD справочников**, либо полноценная
  выгрузка. По `Интеграция_Деканат_и_НагрузкаВУЗа.md` для выгрузки нужны `ExternalId`+`Source` на
  сущностях и upsert — **этого в модели пока нет** (миграции/поля отсутствуют).

## 7. 🟡 Асинхронная генерация (сейчас блокирует HTTP до 180 с)

Солвер (`SolverOptions` = 180 с, 8 воркеров) выполняется синхронно в запросе генерации — UI
«зависает», параллельные запуски съедают CPU. Вынести в фоновую задачу (Channel/`IHostedService`/
Hangfire): `202 Accepted` + id задачи + эндпоинт статуса/прогресса + отмена; ограничить
конкуррентность; `SolverOptions` — в конфиг.
_Файлы: `Application/Solver/Solving/ScheduleSolver.cs`, `GenerateInstituteScheduleCommand.cs`._

## 8. 🟡 Фронтенд (до уровня продукта)

- **vue-router** вместо ручного переключения вкладок (состояние/глубокие ссылки в URL).
- **Стор (Pinia)** для справочников и сессии — сейчас списки перезагружаются на каждой вкладке.
- **Тосты/уведомления** действий (генерация, публикация, CRUD) + «колокольчик» уведомлений нагрузки.
- **Профиль/роль пользователя**, корректный logout, единая обработка истечения токена (401) на всех
  экранах (в `api/http.ts` логика есть — проверить покрытие).
- **Вкладка «Ограничения»**: левые деревья на реальные справочники + редакторы доступности
  преподавателей/аудиторий (сетка) и оснащения оборудованием.
- **Экспорт/печать** расписания.
- Состояния загрузки/скелетоны, единые пустые состояния, базовая адаптивность и a11y.

## 9. 🟢 Безопасность и эксплуатация (минимум)

- **Секреты в `appsettings.json`** (пароль БД, слабый `Jwt:SigningKey`) → env/user-secrets,
  сгенерировать стойкий ключ ≥ 32 байт; в `docker-compose.yml` задать реальные пароли.
- **Swagger и `UseHttpsRedirection` включены всегда** (`if (IsDevelopment)` закомментирован в
  `Program.cs:90-96`) → закрыть Swagger вне Development; за nginx редирект на HTTPS лишний.
- **LDAP без проверки сертификата** (`IgnoreCertificateErrors=true`/`LDAPTLS_REQCERT=never`) — для
  реальной среды включить проверку.
- **Пагинация `/workloads/changes`** (сейчас отдаёт всё) + индекс на `TimeStamp`.
- **`db.Database.Migrate()` на старте** — вынести в отдельный шаг (или оставить только для dev).

## Свод: заявлено в ВКР как готовое, но в коде отсутствует

Транзакции SERIALIZABLE · real-time синхронизация · ролевая модель · сервис уведомлений об
изменении нагрузки · ввод доступности преподавателей/аудиторий из UI · экспорт расписания ·
сопоставление с ММИС по `ExternalId`. Эти пункты стоит реализовать в первую очередь — они и
составляют разницу между «работающим прототипом» и «готовым продуктом».

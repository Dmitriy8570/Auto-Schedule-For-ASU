# Результат генерации расписания (dev-солвер)

Сгенерировано тестом `SemesterScheduleGenerationTests`. Входные данные читаются из
файлов `input/teachers.json`, `input/groups.json`, `input/classrooms.json`. Решение
задаёт расписание на одну неделю, которое повторяется весь семестр.

- Статус: **Feasible**
- Значение целевой функции (сумма штрафов): 3240
- Время решения: 185,67 с
- Переменных в модели: 463344
- Назначено занятий за неделю: 770

## Структура
- `input/` — исходные файлы преподавателей, групп и аудиторий (JSON).
- `00_исходные_данные.md` — сводка входных данных и синтезированных нагрузок.
- `Преподаватели/` — по файлу на преподавателя.
- `Группы/` — по файлу на группу.
- `Аудитории/` — по файлу на аудиторию.

В каждой строке занятия: время, предмет, тип, преподаватель, аудитория и все группы
(для потоковых пар помечено `[ПОТОК]`, для двойных — `[двойная]`).

## Заложенные граничные ситуации
- **Потоковые лекции** на весь курс направления (несколько групп в одной паре).
- **Двойные лабораторные** (электроника) у подгрупп — спаренные соседние слоты.
- **Параллельные подгруппы** (`…/1` и `…/2`) — стремятся в одну пару.
- **Вторая смена** (пары 5–6) и **вечерние группы** (пары 7–8).
- **Требования к оборудованию**: компьютерные классы, лаборатории, проекторы.
- **Предпочитаемый корпус** (мягко) и **переезд между корпусами** (жёстко).
- **Недоступность** преподавателей (пн, пара 1) и аудиторий (лаборатории, сб вечер).

> Замечание dev-модели: студенты родительской группы и её подгрупп моделируются как
> разные субъекты, поэтому пересечение «лекция родителя ↔ лаб. подгруппы» не запрещается
> жёстко. Это известное упрощение текущей версии солвера.

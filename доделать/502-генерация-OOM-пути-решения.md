# Ошибка 502 при автогенерации расписания — причина и пути решения

> Диагностика на 2026-06-22 (живой прогон в Docker). Связано с задачей **T1**.
> Это **не таймаут** (не 504) и **не** проблема асинхронного запуска — асинхронный запуск
> (202 + опрос статуса) и таймауты nginx работают корректно.

---

## 1. Симптом
При запуске автогенерации через некоторое время UI получает **502 Bad Gateway** (на запросах
опроса статуса задачи и на любых `/api/*`).

## 2. Корень проблемы (подтверждён доказательствами)
**Во время работы CP-SAT-солвера процесс бэкенда `dotnet` убивается OOM-killer'ом ядра
WSL2-виртуалки; бэкенд перезапускается, и пока он недоступен, nginx отдаёт 502.**

Цепочка:
1. Генерация строит большую модель (≈ **9918** записей нагрузки × **75** аудиторий × **864** слота).
   Память бэкенда росла наблюдаемо: 0.26 → 1.1 → 2 → 4 → **6.4 ГиБ**.
2. Вся Docker/WSL2-виртуалка имеет **~7.74 ГиБ** ОЗУ на **все 5 контейнеров** (MS SQL ~0.5 ГиБ,
   AD, PostgreSQL, nginx + бэкенд). На ~6.4 ГиБ у бэкенда виртуалка упирается в потолок.
3. Срабатывает **глобальный** OOM-killer ядра и убивает `dotnet`. Из dmesg WSL2:
   ```
   Out of memory: Killed process (dotnet) total-vm:41705232kB, anon-rss:6738944kB ... global_oom
   oom-kill: constraint=CONSTRAINT_NONE ... task=dotnet
   ```
   `global_oom` / `constraint=CONSTRAINT_NONE` = это OOM **всей виртуалки**, а не контейнерного
   cgroup-лимита (поэтому `docker inspect … OOMKilled=false`).
4. Docker по политике `restart: unless-stopped` поднимает бэкенд (`RestartCount` рос 1→2).
   Пока он стартует — nginx не может достучаться до upstream и отдаёт **502**.

### Почему текущие настройки усугубляют
- `docker-compose.yml` → backend `deploy.resources.limits.memory: **8G**`, а сама виртуалка
  **~7.74 ГиБ**. Лимит 8 ГиБ **недостижим** → контейнерный cgroup-OOM (мягкий, со статусом
  `OOMKilled`) **не срабатывает**, первым «выстреливает» OOM ядра виртуалки (жёстко, убивает процесс).
- `appsettings.json` → `Solver.SearchWorkers: **8**`. CP-SAT в портфельном режиме держит на каждый
  воркер своё состояние поиска → пиковая память во время решения растёт с числом воркеров.
- Хост: **16 ГБ** физических, WSL2 по умолчанию берёт ~50% (≈8 ГБ). Файла `.wslconfig` нет —
  значит память виртуалки **можно поднять** (запас на хосте есть).

---

## 3. Пути решения

Сгруппированы: (A) быстрые обходные без кода, (B) снижение аппетита солвера, (C) устойчивость
(чтобы 502 не было даже при нехватке памяти). Можно комбинировать.

### Группа A. Инфраструктура / конфиг (быстро, без изменения логики)

**A1. Поднять память WSL2-виртуалки.** ← самый быстрый, на этом хосте есть запас (16 ГБ).
Создать `C:\Users\barda\.wslconfig`:
```ini
[wsl2]
memory=12GB
swap=4GB
```
Затем `wsl --shutdown` и перезапустить Docker Desktop. Проверить: `docker info` → `Total Memory`
должно стать ~12 ГБ. Оставляет 4 ГБ хосту под Windows.
- ➕ ноль изменений в коде; даёт солверу запас.
- ➖ упирается в физический потолок хоста; не масштабируется на большие вузы.

**A2. Исправить лимит памяти контейнера** в `docker-compose.yml` (сейчас вводит в заблуждение).
Выставить лимит **ниже** размера виртуалки (напр. при `.wslconfig` 12 ГБ — backend `limits.memory: 9G`,
`reservations.memory: 4G`). Тогда при переборе сработает **контейнерный** OOM (предсказуемо, со
статусом `OOMKilled=true`, не валит соседние контейнеры MS SQL/AD).
- ➕ контролируемое падение, не страдают другие сервисы.
- ➖ сама генерация всё равно не завершится — просто падает «чисто».

**A3. Снизить число воркеров поиска.** `backend-auto-schedule/src/API/appsettings.json` →
`Solver.SearchWorkers: 8` → **4** (или **2**). Уменьшает пиковую память фазы решения.
- ➕ одна строка, заметно режет пик во время поиска.
- ➖ меньше параллелизма поиска (на этом масштабе решение и так Unknown); не уменьшает память
  **построения** модели (базовый массив переменных строится один раз — см. группу B).
- ⚠️ асинхронный эндпоинт сейчас не принимает `searchWorkers` (только синхронный, query-параметром) —
  поэтому менять именно в `appsettings.json`.

### Группа B. Снизить размер самой модели (правильное решение для масштаба)

Базовая память — это сама модель `BoolVar[нагрузки × аудитории × слоты]` и ограничения над ней;
строится один раз и не зависит от числа воркеров. Чтобы памяти хватало на реальном масштабе:

**B1. Более мелкая декомпозиция.** Сейчас генерация идёт по институту (≈3300 нагрузок) — модель всё
ещё огромна. Декомпозировать мельче: по **степени/курсу/потоку**, решать порциями с фиксацией уже
занятых ресурсов (механизм `OccupiedResources` / `ScheduleDataProvider` уже есть — расширить на
меньшие единицы). Меньше нагрузок за прогон → кратно меньше переменных и памяти.
- ➕ единственный способ устойчиво масштабироваться; ускоряет и качество решения.
- ➖ изменение алгоритма генерации и слияния результатов; тесты.

**B2. Усилить прунинг модели** в `StaticFeasibility`/`VariablesSectionBuilder`. Сейчас отсекаются
тройки по вместимости/оборудованию/смене. Добавить отсечения: привязка нагрузки к корпусам своего
института/кафедры (резко режет «аудитории»), более раннее ограничение слотов по смене и т. п.
Меньше `BoolVar` → меньше памяти и ограничений.
- ➕ уменьшает базовую модель без смены архитектуры.
- ➖ нужно аккуратно, чтобы не отсечь осуществимые варианты (риск Infeasible).

**B3. Параметры CP-SAT по памяти** (`StringParameters`): ограничить рост (напр. умеренный
`max_memory_in_mb`, если поддерживается версией OR-Tools) — как «страховка», не как основное решение.

### Группа C. Устойчивость — чтобы 502 не было в принципе

**C1. Изолировать солвер в отдельный процесс/воркер.** Тяжёлую генерацию запускать дочерним
процессом (или отдельным сервисом-воркером). Тогда OOM убьёт **только воркер**, а API/бэкенд
останутся живы → **никакого 502**, а задача честно помечается `Failed`. Это разрывает связь
«падение солвера = падение всего API».
- ➕ устраняет сам симптом 502 при любой нехватке памяти; правильная прод-архитектура.
- ➖ заметная переработка (межпроцессное взаимодействие, передача результата).

**C2. Понятный статус вместо 502.** На фоне C1 — отдавать в UI статус задачи «Failed: недостаточно
памяти», вместо того чтобы опрос статуса падал в 502. (Сейчас при падении всего бэкенда даже опрос
статуса возвращает 502, т.к. upstream недоступен — поэтому без изоляции (C1) одним UI это не лечится.)

---

## 4. Рекомендуемый порядок
1. **A1** (поднять WSL2 до 12 ГБ) + **A2** (реалистичный лимит контейнера) — снимает проблему «здесь
   и сейчас» на dev-хосте 16 ГБ, без кода.
2. **A3** (воркеры 8→4) — дешёвое снижение пика.
3. **C1** (изоляция солвера) — чтобы 502 не возникал даже при нехватке памяти (устранение симптома).
4. **B1/B2** (мельче декомпозиция + прунинг) — устойчивое решение для реального масштаба вуза.

## 5. Быстрая проверка после изменения
1. `docker info` → `Total Memory` (после A1 ~12 ГБ).
2. Запустить генерацию института и параллельно следить:
   `docker stats auto-schedule-backend` (память не должна упираться в потолок виртуалки) и
   `docker inspect auto-schedule-backend --format '{{.RestartCount}} {{.State.OOMKilled}}'`
   (RestartCount не должен расти).
3. Опрос статуса задачи не должен отдавать 502; задача доходит до `Succeeded`/`Failed`.
4. (При наличии доступа) в WSL: `dmesg | grep -i oom` — новых `Killed process (dotnet)` быть не должно.
